================================================================================
  1. EXECUTIVE SUMMARY
================================================================================

Beit V'Metaplim is a community portal connecting trauma patients (PTSD, combat
veterans, Nova survivors) with volunteer therapists offering subsidized/free
mental health care. The system consists of:

  - PUBLIC WEBSITE: Static HTML/JS site with Tailwind CSS, deployed on Vercel
  - LEARNING PORTAL: Free NLP practitioner courses with video player, notes,
    AI tutor chatbot, and progress tracking
  - ADMIN DASHBOARD: Full CRM with 5 data tables, GA4 analytics, settings,
    and WhatsApp bot management
  - CRM BOT: WhatsApp-based business automation (Express.js on Fly.io)
  - FINANCE BOT: Personal finance tracker with receipt OCR (Express.js)
  - DATABASE: Supabase (PostgreSQL) with 15+ tables and 40+ RLS policies
  - EDGE FUNCTIONS: AI tutor (Claude Sonnet 4.5) + GA4 analytics aggregator

The platform serves ~350+ therapists and processes leads from marketing
campaigns. All UI is in Hebrew (RTL). The architecture prioritizes lead
capture (no login required to submit forms) and security (RLS-first design).

================================================================================
  6. ROW LEVEL SECURITY (RLS) POLICIES
================================================================================

  TABLE: profiles
  ----------------------------------------------------------------
  Policy                          | Op     | Condition
  --------------------------------|--------|------------------------
  Users can view own profile      | SELECT | auth.uid() = id
  Users can update own profile    | UPDATE | auth.uid() = id
  Admins can view all profiles    | SELECT | profiles.role = 'admin'
  Admins can update all profiles  | UPDATE | profiles.role = 'admin'
  Admins can delete profiles      | DELETE | profiles.role = 'admin'

  TABLE: patients
  ----------------------------------------------------------------
  Anyone can submit patient form  | INSERT | true (lead capture!)
  Patients can view own record    | SELECT | auth.uid() = user_id
  Admins can view all patients    | SELECT | profiles.role = 'admin'
  Admins can update patients      | UPDATE | profiles.role = 'admin'
  Admins can delete patients      | DELETE | profiles.role = 'admin'
  Therapists see assigned patients| SELECT | via matches table join

  TABLE: therapists
  ----------------------------------------------------------------
  Anyone can submit application   | INSERT | true (lead capture!)
  Therapists view own record      | SELECT | auth.uid() = user_id
  Therapists update own record    | UPDATE | auth.uid() = user_id
  Admins can view all             | SELECT | profiles.role = 'admin'
  Admins can update               | UPDATE | profiles.role = 'admin'
  Admins can delete               | DELETE | profiles.role = 'admin'

  TABLE: contact_requests
  ----------------------------------------------------------------
  Anyone can submit               | INSERT | true (lead capture!)
  Admins can view all             | SELECT | profiles.role = 'admin'
  Admins can update               | UPDATE | profiles.role = 'admin'
  Admins can delete               | DELETE | profiles.role = 'admin'

  TABLE: legal_consents
  ----------------------------------------------------------------
  Users can view own consents     | SELECT | auth.uid() = user_id
  Users can insert own consent    | INSERT | auth.uid() = user_id
  Admins can view all             | SELECT | profiles.role = 'admin'

  TABLE: appointments
  ----------------------------------------------------------------
  Patients view own               | SELECT | patient_id -> patients.user_id
  Therapists view own             | SELECT | therapist_id -> therapists.user_id
  Admins manage all               | ALL    | profiles.role = 'admin'

  TABLE: user_notes
  ----------------------------------------------------------------
  Users manage own notes          | ALL    | auth.uid() = user_id

  TABLE: ai_chat_usage
  ----------------------------------------------------------------
  Users read own usage            | SELECT | auth.uid() = user_id
  Users insert own                | INSERT | auth.uid() = user_id
  Users update own                | UPDATE | auth.uid() = user_id

  TABLE: crm_bot_phones

  10. AUTHENTICATION & AUTHORIZATION FLOWS
================================================================================

10.1 USER LOGIN FLOW
----------------------
  1. User navigates to protected page
  2. auth-guard.js calls checkAccess({ requireAuth: true })
  3. No session -> toast "יש להתחבר" -> redirect to /pages/login.html
     (saves returnUrl to sessionStorage)
  4. Login page offers:
     a) Email/password form -> Auth.signIn()
     b) Google OAuth button -> Auth.signInWithGoogle()
        -> Google consent screen -> callback with #access_token
  5. On success: ensureProfile() creates profiles row if missing
  6. Auth state listener fires -> UI updates
  7. Redirect by role or to returnUrl

10.2 LEGAL CONSENT FLOW
-------------------------
  1. User logged in but no legal_consents record
  2. auth-guard.js detects missing consent
  3. Redirect to legal-gate.html
  4. User scrolls through terms (scroll-lock)
  5. Checkboxes unlock at bottom (terms, age 18+, not in emergency)
  6. Submit -> signLegalAgreement() captures:
     - user_id, agreed_version ('1.0'), ip_address (ipify.org), user_agent
  7. Redirect to dashboard

10.3 ADMIN VERIFICATION
-------------------------
  1. Admin page loads
  2. checkAccess({ requiredRole: 'admin' })
  3. Fetches profiles.role from Supabase
  4. If role !== 'admin' -> redirect to role-appropriate dashboard
  5. Admin dashboard renders with full CRM access

10.4 LEAD CAPTURE (No Auth)
-----------------------------
  1. Anonymous visitor fills patient-step1-4 or therapist-step1-4
  2. Form data submitted directly to Supabase via anon key
  3. RLS INSERT policy = true (open to anyone)
  4. Data saved to patients/therapists/contact_requests
  5. CRM bot polls for new leads every 2 minutes
  6. Admin notified via WhatsApp DM

12.1 STRENGTHS
---------------
  [+] RLS-first architecture: all tables protected by Row Level Security
  [+] Admin verified via database role (profiles.role), not hardcoded emails
  [+] Legal consent tracked with IP + user agent + timestamp for audit
  [+] OAuth race condition handled properly in auth-guard.js
  [+] CRM bot uses IP whitelist + token + phone authorization (3 layers)
  [+] Webhook returns 200 immediately (prevents retry spam)
  [+] Hebrew error translations (prevents info leakage of Supabase internals)
  [+] Form autosave to localStorage (prevents data loss, not a security feature)
  [+] CORS whitelist on CRM bot (not wildcard *)
  [+] GA4 Edge Function whitelists specific origins
  [+] Service-role key only on server-side (never in frontend)
  [+] Anon key in frontend is by design (RLS protected)
  [+] Cookie consent banner before analytics load (GDPR compliant)
  [+] Rate limiting on CRM bot (100 req/15min global, 60 req/min webhook)
  [+] Graceful degradation: AI fallback -> keyword intent if Groq/Claude down
  [+] .gitignore includes .env files

12.2 VULNERABILITIES & RISKS
------------------------------

  CRITICAL:
  [!] Verify .env files have NEVER been committed to Git history.
      If they have, ALL API keys (Green API, Supabase service-role,
      Anthropic, Groq, Gemini) must be rotated immediately.
      Check: git log --all -- .env

  HIGH:
  [!] No rate limiting on frontend forms (patient-step, therapist-step).
      Attacker could flood database with thousands of fake lead records.
      Fix: Add Supabase Edge Function rate limiter or client-side throttle.

  [!] No rate limiting on login form.
      Brute-force attack possible on email/password.
      Fix: Max 5 attempts per 15 minutes per IP + CAPTCHA after 3 failures.

  [!] Insufficient server-side input validation.
      Forms rely on HTML5 attributes only (no backend validation).
      No maxlength on textarea fields (could submit 10MB+ strings).
      Fix: Add Supabase database CHECK constraints + Edge Function validation.

  MEDIUM:
  [!] No Content Security Policy (CSP) headers.
      XSS attacks could inject scripts if DOM manipulation is unsafe.
      Fix: Add CSP meta tag or Vercel headers config.

  [!] No X-Frame-Options header.
      Site could be framed by malicious sites (clickjacking).
      Fix: Add X-Frame-Options: SAMEORIGIN in vercel.json headers.

  [!] No CSRF protection on forms.
      Mitigated by Supabase JWT requirement, but authenticated users
      could be CSRF'd into unwanted form submissions.
      Fix: Add anti-CSRF tokens to admin dashboard forms.

  [!] IP capture via external API (ipify.org) for legal consent.
      Free API could be rate-limited, spoofed, or unavailable.
      Fix: Migrate to backend-captured IP (Edge Function or X-Forwarded-For).

  [!] CRM bot crm_bot_access defaults to 'admin' if no row exists.
      New users without explicit role assignment get full admin access.
      Fix: Default to 'viewer' and require explicit admin grant.

  [!] Finance bot cost-tracker endpoint (/api/cost-event) has no auth.
      Anyone who knows the URL can POST fake cost events.
      Fix: Add shared secret or JWT verification.

  LOW:
  [!] No consent withdrawal mechanism (GDPR may require it).
  [!] Console logs may contain PII (phone numbers, names, therapy concerns).
  [!] Webhook phone number not regex-validated before use in CRM bot.
  [!] http:// origins in CRM bot CORS config (dev-only, remove for prod).

13.2 ARCHITECTURAL TECHNICAL DEBT
-----------------------------------

  1. NO BUILD SYSTEM
     The frontend is pure static HTML/JS with no bundler (Webpack, Vite, etc.).
     Pros: Simple deployment, no build step, fast iteration.
     Cons: No tree-shaking, no minification, no code splitting, no TypeScript.
     Impact: Larger page loads, no compile-time error catching.

  2. GLOBAL WINDOW NAMESPACE
     All modules export to window.* (window.Auth, window.Profiles, etc.).
     Pros: Simple inter-module communication.
     Cons: No module isolation, potential naming conflicts, hard to test.
     Impact: Technical debt grows with each new module.

  3. SUPABASE-CLIENT.JS IS A MONOLITH (988 lines)
     Single file handles auth, profiles, patients, therapists, appointments,
     course progress, notes, certifications, UI, form autosave, retry logic.
     Recommendation: Split into separate ES modules when build system is added.

  4. NO AUTOMATED TESTING
     No unit tests, integration tests, or E2E tests.
     CRM bot has no test suite. Frontend has no test suite.
     Risk: Regressions on every change. Manual QA only.

  5. NO ENVIRONMENT SEPARATION
     Same Supabase instance for dev and production.
     CRM bot .env shared across local dev and Fly.io deployment.
     Risk: Dev mistakes affect production data.

  6. SQLITE IN FINANCE BOT (No Backup)
     Finance bot uses local SQLite -- no backup policy, no replication.
     If Fly.io machine dies, all financial data is lost.
     Fix: Migrate to Supabase or add scheduled SQLite backup to cloud storage.

  7. SINGLE INSTANCE DEPLOYMENT
     Both bots run on single Fly.io machines (no redundancy).
     If machine goes down, bot is offline until restart.
     Fix: Set min_machines_running > 1 for high availability.

  8. NO MONITORING/ALERTING INFRASTRUCTURE
     No Sentry, DataDog, or similar error tracking.
     Errors only visible in Fly.io logs (7-day retention).
     Fix: Add error tracking service + uptime monitoring.

  9. DUPLICATED AUTH LOGIC
     auth-guard.js and supabase-client.js both handle auth state.
     Profile creation logic (ensureProfile) runs on every page load.
     Opportunity: Consolidate into single auth module.

  10. NO API VERSIONING
      Edge Functions have no version prefix (/v1/ai-chat).
      Breaking changes would affect all clients simultaneously.
      Low priority: Only 2 Edge Functions currently.
