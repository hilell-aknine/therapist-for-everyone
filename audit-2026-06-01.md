**Method:** 5 parallel deep-scan agents (Supabase schema / Popups / WhatsApp+Automation / AI Teacher / Lessons+Game)

## 0. תקציר מנהלים (Hebrew TL;DR — for Hillel)

האתר בנוי כ-**HTML/JS סטטי + Supabase** (אין React, אין "components/"). סרקתי 5 מערכות לעומק. הממצא המרכזי לשיפור **שיעור ההשלמה (Completion Rate)**:

1. **אין מקור-אמת אחד להשלמה.** התקדמות וידאו נשמרת קודם ב-localStorage ורק אחר כך ב-Supabase (fire-and-forget) — כך שמשתמש לא-מחובר או שנכשל לו הסנכרון **לא נספר בכלל**. בנוסף, התקדמות הקורס (`course_progress`) והתקדמות המשחק (`nlp_game_players`) הן **שתי מערכות נפרדות שלא מדברות זו עם זו**.
2. **הפופאפים של "סיום שיעור" שבורים בחצי מהאתר.** הדף החדש (`course-library-v2.html`) לא קורא ל-`notifyLessonComplete()`, ולחיצות (clicks) נרשמות רק לפופאפ אחד — כך שמדדי ה-CTR בלוח הניהול **מעוותים**.
3. **המורה ה-AI שורף טוקנים.** בסיס ידע של ~65KB (כ-17 אלף טוקנים) נשלח **בכל הודעה**, ללא סינון לפי שיעור וללא caching. אין תיעוד עלות/טוקנים בכלל.
4. **WhatsApp = שגר-ושכח ללא retry.** אם Green API נכשל — ההודעה אובדת בשקט, אין לוג כשלים ואין התראה. ה-webhook ל-CRM נמחק (pg_net) ומסתמך כעת על polling של בוט חיצוני שאם נופל — הכל שותק.
5. **הנתונים אינם "real-time אמיתי".** מוני-עזר (lesson_count, lessons_completed) הם denormalized וזולגים; אין `updated_at` אוטומטי; אין היסטוריית ניסיונות — רק "מצב אחרון".

**שורה תחתונה:** בלי כתובת אחת אמינה להשלמה (DB-first + טבלת אירועים), כל דשבורד Completion-Rate שנבנה יהיה מבוסס על נתונים חלקיים.

## 2. DATABASE SCHEMA (Supabase)

### 2.1 Tables relevant to completion analytics

| Table | Source migration | Purpose | RLS | Key columns |
|---|---|---|---|---|
| `course_progress` | `sql/007` + `047` (indexes) + `20260315180000` (RLS) | Video watch + completion | ✅ (no DELETE) | `user_id`→auth.users, `video_id` TEXT, `course_type`, `lesson_number`, `completed`, `completed_at`, `watched_seconds`, `updated_at` |
| `lessons` | `028` + `029` (seed 51 rows) | Lesson catalog (metadata only) | ✅ | `id` UUID, `title`, `video_url` (full URL), `section_name`, `order_index` |
| `courses` | `sql/create_courses_table` | Course catalog | ✅ | `slug` UNIQUE, `lesson_count` (denormalized!), `status` |
| `course_access` | `sql/create_course_access` | Per-user locked-course grants | ✅ | `user_id`, `course_slug`→courses.slug |
| `user_notes` | `008` | Per-lesson notes | ✅ (full CRUD) | `user_id`, `video_id`, `course_type`, `content`, `updated_at` |
| `nlp_game_leaderboard` | `042` | Game aggregate scores (1 row/user) | ✅ | `user_id` UNIQUE, `total_xp`, `lessons_completed` (denormalized!), `current_streak`, `last_active` |
| `nlp_game_players` | `20260503140000` | Full game state (1 row/user) | ✅ (no DELETE) | `user_id` PK, `completed_lessons` JSONB, `module_accuracy` JSONB, `xp`, `streak`, counters |
| `ai_chat_usage` | `sql/006` + `053` | Daily message rate-limit counter | ✅ (no admin SELECT) | `user_id`, `message_count`, `date`, `source` |

**Indexes present:** `course_progress` has `user_id`, `(user_id,video_id)`, `(user_id,course_type)`, partial `completed_at DESC WHERE completed`. `lessons` has `(section_name,order_index)`. `user_notes` has `user_id`, `(user_id,updated_at)`. `nlp_game_leaderboard` has `total_xp DESC`, `user_id`, `updated_at DESC`.

### 2.2 Missing FKs / indexes (analytics impact)

| Gap | Severity | Impact on real-time completion data |
|---|---|---|
| **No join key between `course_progress.video_id` (bare YouTube ID) and `lessons.video_url` (full URL)** | 🔴 High | Per-section/per-module completion requires a non-indexable `LIKE '%'||video_id` expression scan. No FK integrity. This is the single biggest schema blocker for accurate funnel analytics. |
| `course_progress.course_type` has no FK to `courses.slug` | 🟠 Med | Free-text; orphan rows on rename; `GROUP BY course_type` runs on unvalidated text. |
| `courses.lesson_count` is a hardcoded denormalized counter (no trigger) | 🔴 High | Completion % uses `totalVideos` passed from JS, not derived from DB. Drifts silently when lessons are added → wrong %. |
| No `(user_id, completed)` partial index matching the admin RPC's `WHERE completed=true AND video_id NOT LIKE 'last_watched_%'` | 🟠 Med | `admin_segments_overview` rescans `course_progress` every call; slows as users grow. |
| `nlp_game_players.completed_lessons` JSONB has no GIN index | 🟠 Med | "How many completed module 1" = full-table JSONB scan. |
| `user_notes` missing `(user_id, course_type, updated_at)` composite | 🟢 Low | Multi-course note queries post-filter. |

### 2.3 Schema blind spots

- **`last_watched_*` synthetic rows** live in `course_progress` (storing resume position, `completed=false`, abusing `watched_seconds` as moduleIndex). **Every** analytics query must `AND video_id NOT LIKE 'last_watched_%'` or counts are wrong. Only the admin RPC remembers this.
- **No `updated_at` auto-trigger** on `course_progress`, `nlp_game_*`, `user_notes` — "last active" timestamps go stale on UPDATE (client sometimes sets it manually, sometimes not).
- **No event log — only latest state.** UPSERT on `(user_id, video_id)` means re-watches overwrite. Cannot answer "when first completed," "how many attempts," cohort progression over time.
- **Denormalized counters drift:** `courses.lesson_count`, `nlp_game_leaderboard.lessons_completed` vs `nlp_game_players.completed_lessons`, `perfect_lessons` (int) vs `perfect_lessons_list` (jsonb) — all updated by separate JS paths, no trigger keeps them in sync.

## 3. POPUP SYSTEM

### 3.1 Files
`js/popup-manager.js` (engine) · `js/admin/admin-popups.js` (CRUD + funnel stats) · `pages/popup-preview.html` · `docs/popup-insights.md` · consumers `pages/course-library.html` and `pages/course-library-v2.html`. Schema: migrations `044, 045, 054, 055, 056, 057` + `20260419110000` (tables `popup_configs`, `popup_events`, `popup_dismissals`, `popup_insights_log`).

### 3.2 Lesson-completion trigger logic (the actual chain)

**course-library.html (primary):**
`selectLesson()` arms a 45s `setTimeout` → `markLessonCompleted(m,l)` → (a) push to in-memory `completedLessons` + `localStorage`; (b) `CourseProgress.markVideoWatched()` → Supabase UPSERT (only if authenticated); (c) if `completedLessons.length >= 3` → direct `PopupManager.request('share_prompt')`; (d) `PopupManager.setLessonCount()`; (e) `PopupManager.notifyLessonComplete()` → `scanTriggered('lesson_complete')` → iterates `popup_configs` where `trigger_event='lesson_complete' AND is_active AND status='live'` → `publicRequest()` → queue checks (audience, `trigger_min_lessons`, cooldown, daily cap) → `showPopup()` → `logEvent(id,'shown')`.

**So:** completion is a **client-side localStorage event**, NOT a DB read. The DB write is a side effect that can fail without blocking the popup. `trigger_min_lessons` is checked against `userContext.lessonCount` (from the localStorage array).

### 3.3 Event tracking
`logEvent(popupId, eventType)` inserts into `popup_events` `{popup_id, event_type, user_id | session_id, variant, created_at}`. `shown` fires automatically in `showPopup()`. `dismissed`/`clicked` fire **only when page code explicitly calls** `PopupManager.dismiss()` / `.click()`. `session_id` from `sessionStorage` (lost on tab close); if unavailable, anonymous events are **dropped silently**.

### 3.4 Popup blind spots

| Gap | Severity | Detail |
|---|---|---|
| **`course-library-v2.html` never calls `notifyLessonComplete()` or `setLessonCount()`** | 🔴 High | Any DB popup with `trigger_event='lesson_complete'` **never fires on v2**; `trigger_min_lessons` gates always block (lessonCount stays 0). v2 uses its own ad-hoc `request()` calls. |
| **`clicked` only wired for `share_prompt`** | 🔴 High | `auth_modal` (the key conversion popup) CTR is **permanently 0** in the data → admin dashboard CTR is misleading. |
| `popup-manager.js` only loaded on 2 pages | 🔴 High | `nlp-game.html`, `master.html`, `free-portal.html` etc. never fire configured popups. |
| `video_toast` close uses `close()` not `dismiss()` | 🟠 Med | No `dismissed` event logged → dismiss-rate = 0. |
| No `lesson_id` / `lesson_number` column on `popup_events` | 🟠 Med | Cannot answer "which lesson triggered the popup" without time-proximity guessing against `course_progress`. |
| `timeout` event type defined but never fired | 🟢 Low | Dead column + dead dashboard counter. |
| `lessonCount` stale at login (read before `mergeProgress()`) | 🟠 Med | Cross-device completions under-count → gated popups wrongly blocked. |

## 4. WHATSAPP & AUTOMATION

### 4.1 Inventory
**Welcome:** `supabase/functions/send-welcome-whatsapp/index.ts` — Green API portal instance (`GREEN_API_INSTANCE`/`GREEN_API_TOKEN`, no hardcoded ID). **Retention:** `retention-run` generates Claude-drafted messages as `status='draft'`; **actual send is the external CRM bot, not Supabase.** Dead path: `032/033` installed a `pg_net` webhook to the CRM bot → **dropped** in `20260315153432` (pg_net unavailable).

### 4.2 Lifecycle (welcome)
5 client call sites all use `fetch(...).catch(()=>{})` (fire-and-forget): `free-portal.html`, `portal-questionnaire.html`, `login.html`, `login-v2.html`, and the catch-all `ensureProfile()` in `supabase-client.js` (runs every page load when phone present & `whatsapp_welcome_sent_at IS NULL`). Idempotency via `profiles.whatsapp_welcome_sent_at` + `portal_questionnaires.welcome_sent_at` + rollout-cutoff dates.

### 4.3 Error handling — the good and the gap
**Good:** `markSent()` is called **only after** confirmed Green API `result.ok` (no before-send flag → no false "sent"). `checkOk=false` returns 503 without marking → retryable.
**Gap:** every failure (502/503/500) is swallowed by the client `.catch(()=>{})`. The **only** retry is `ensureProfile()` firing on a future page load — which works for the profile path but **not** the questionnaire path. A user who fails once and never returns **never gets the welcome**.

### 4.4 Automation blind spots

| Gap | Severity | Detail |
|---|---|---|
| Fire-and-forget, no retry queue, no failed-send log | 🔴 High | No `send_error` column on profiles/questionnaires, no admin alert. Failures invisible except in ephemeral Edge logs. |
| All 34 `bot_automation_configs` crons + `automation_rules` execute **only inside the Fly.io CRM bot** | 🔴 High | If the bot machine is down/redeployed, ALL automations (patient_welcome, new_lead_polling, reminders) go silent. No dead-man's switch, no Supabase-side health check. |
| `welcome_sent_at` is semantically overloaded | 🟠 Med | Set on success, not-on-WhatsApp, opt-out, AND pre-rollout. `IS NOT NULL` ≠ "delivered." |
| Subscription crons in CLAUDE.md don't exist | 🟢 Low | `subscriptions` table archived (`20260517100000`); the 3 documented crons have no migrations. |

## 5. AI TEACHER / CHAT

### 5.1 Two backends
- **`ai-chat`** (portal, free + paid Master) → **OpenRouter**, model `stepfun/step-3.5-flash:free`, `OPENROUTER_API_KEY`, `max_tokens: 4096`, no fallback. Hosted in `course-library.html` (free tab `sendAiMessage`, Master tab `sendMasterAiMessage`).
- **`gemini-mentor`** (the NLP game) → **Gemini `gemini-2.0-flash`** (`GEMINI_API_KEY`) → OpenRouter fallback (`stepfun` → `nvidia/nemotron-3-nano`), `maxOutputTokens: 500`, single-turn (no history).

### 5.2 Token management
History windowed to last 10 turns (`slice(-10)`) in the portal; game sends no history. Daily caps enforced server-side via `ai_chat_usage` (200/day `chat`, 100/day `mentor`). **`ai_chat_usage` logs only `message_count` — NO prompt/completion token columns, NO cost.** Zero cost visibility.

### 5.3 Context passing & token bloat — pinpointed

| # | Bloat | Location | Per-call cost |
|---|---|---|---|
| 1 | **Full 65 KB knowledge base baked into the system prompt of EVERY message**, unfiltered by lesson, uncached | `ai-chat/knowledge.ts` (66,659 bytes / ~840 lines) → embedded in `SYSTEM_PROMPT` at `ai-chat/index.ts:28` | **~16,000–18,000 input tokens every turn** |
| 2 | No token/cost logging | `ai_chat_usage` schema | No visibility; `data.choices[].usage` never read |
| 3 | `max_tokens: 4096` on conversational Q&A (8× the game's 500) | `ai-chat/index.ts:174` | latency/cost ceiling |
| 4 | Full 10-turn history re-sent after the giant prompt | `course-library.html` send funcs | grows to ~22k+ tokens mid-session |
| 5 | Blocking `setTimeout(3000)` on Gemini 429 retry holds the Deno isolate | `gemini-mentor/index.ts:51-55` | timeout cascades under load |

**Root cause:** `knowledge.ts` is a module-level constant concatenated into the system prompt with no lesson-scoping and no prompt-cache. The lesson "context" the client sends is only a **title string** — the actual 20-minute transcript is never passed; the model relies entirely on the static blob.

## 6. LESSON STRUCTURE & GAME DATA

### 6.1 Three parallel lesson systems (drift risk 🔴)
- **A — Supabase `lessons` table** (51 rows, metadata only) served by `get-lessons` Edge Function. **No page actually calls `get-lessons`** — it's an unwired stub. `course-library.html` uses **hardcoded `masterModules`/`masterLessons` arrays** instead.
- **B — Static JS** `js/nlp-game-data-m1..m7.js` → assembled into `MODULES` by `nlp-game-data.js`. This is what the **game actually reads** (in-memory, no network).
- **C — Static transcripts** `docs/transcripts/per-lesson/*.txt` (51 files) = authoring source, manually converted into B.

Same 51 lessons defined independently in A and B; titles duplicated verbatim; change one → silent divergence.

### 6.2 Game data shape
7 modules / 51 lessons (M1:8, M2:6, M3:7, M4:7, M5:7, M6:8, M7:8). Each lesson: `reading.paragraphs[] + keyTerms[]` and `exercises[]` (8 types, `correct` index, `wrongExplanations[]` with null at correct). XP is hardcoded in the **engine**, not the data: +50/lesson, +50/daily-challenge, +100/story, incremental per correct answer.

### 6.3 Read / save paths
- **Read:** `nlp-game.html` loads m1–m7 → `MODULES`; `game.startLesson()` reads `currentModule.lessons[i]` from memory. No network.
- **Save:** `savePlayerData()` writes localStorage synchronously + debounced (1s) `saveToSupabase()` → **`.update()` (not upsert)** on `nlp_game_players` (1 retry after 2s). On `completeLesson()`, `NLPLeaderboard.syncScore()` **upserts** `nlp_game_leaderboard` on `user_id`. **Per-completion, never per-attempt.**

### 6.4 Game blind spots

| Gap | Severity | Detail |
|---|---|---|
| **Game progress (`nlp_game_players.completed_lessons` keyed `"m-l"`) and video progress (`course_progress` keyed YouTube ID) are unlinked** | 🔴 High | No shared FK/mapping → cannot unify "did this user complete this lesson" across video + game. |
| Full-state overwrite, no per-attempt history | 🔴 High | Only final cumulative state in DB; no `completed_at` per lesson (flat boolean map); can't reconstruct timeline/attempts. |
| Guest (unauthenticated) play never reaches Supabase | 🔴 High | Large slice of engagement = zero analytics. |
| `nlp_game_players` didn't exist before `20260503140000` | 🟠 Med | Pre-2026-05-03 players' progress lives only in browser localStorage. |
| `module_accuracy` JSONB may never be populated | 🟠 Med | No writes to `playerData.moduleAccuracy` found → likely always `{}`. |
| Weekly leaderboard is `updated_at`-gated, not `xp_delta` | 🟢 Low | Old XP appears "weekly" if user touched game this week. |

## 7. Cross-cutting verdict: why data is NOT 100% real-time accurate

1. **localStorage-first, DB-second, fire-and-forget.** The authoritative completion event happens in the browser; the DB write is best-effort and silently queued to `pendingSync` on failure (a queue with no server mirror). Offline / guest / failed-sync completions are invisible.
2. **Two disjoint progress universes** (`course_progress` for video, `nlp_game_players` for game) with **no join key**, plus a third unwired `lessons` table.
3. **Latest-state-only model** (UPSERT, no event tables) on both progress and popups → no timeline, no attempts, no cohort progression.
4. **Denormalized counters with no triggers** → drift between displayed % and reality.
5. **Instrumentation holes:** popup `clicked` only on one popup; v2 page bypasses the DB trigger layer; `popup_events` lacks lesson linkage; AI usage lacks token/cost; WhatsApp lacks a failure log.

## 8. Prioritized remediation backlog (fix-package sized)

**P0 — Make completion a single source of truth**
1. Add `lessons.video_id` (bare ID, generated/normalized) + backfill, then add `course_progress.lesson_id UUID REFERENCES lessons(id)` and write it on completion. Enables real FK joins + per-section funnel.
2. Replace fire-and-forget DB write with a confirmed write before firing analytics; mirror the `pendingSync` queue to a `progress_sync_failures` table so offline/failed completions are visible.
3. Create `lesson_completion_events` (append-only: user_id, lesson_id, source ∈ {video,game}, completed_at, session_id) — one event log feeding all dashboards. Unifies video + game.

**P1 — Fix popup/event instrumentation (cheap, high-leverage)**
4. In `course-library-v2.html`: call `PopupManager.setLessonCount()` + `notifyLessonComplete()` in `markLessonCompleted()`.
5. Wire `PopupManager.click()` on all conversion popups (esp. `auth_modal`); switch `video_toast` close to `dismiss()`.
6. Add `lesson_number`/`lesson_id` to `popup_events`.

**P1 — AI token diet**
7. Lesson-scope the knowledge: pass only the relevant module/lesson slice instead of the full 65 KB; OR move to an Anthropic model with `cache_control` prompt caching on the static blob. Add `prompt_tokens`/`completion_tokens`/`cost` columns to `ai_chat_usage` from the provider `usage` object.

**P2 — Automation resilience**
8. Add `send_error`/`failed_at` columns + an admin "failed welcomes" view; add a Supabase-side heartbeat/alert if the CRM bot stops (dead-man's switch).

**P2 — Schema hygiene**
9. `updated_at` BEFORE UPDATE triggers on `course_progress`, `nlp_game_*`, `user_notes`. Trigger to keep `courses.lesson_count` = `COUNT(lessons)`. Partial index matching the admin RPC. GIN index on `completed_lessons` (or migrate to the event table in #3).
